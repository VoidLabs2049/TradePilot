# ETF 全天候 Forward Evidence 与 Live Pilot 准入设计

## 目的

本文定义 ETF 全天候策略在完成 Stage O 工程闭环后，如何用 3～6 个真实时间调仓周期评估数据、决策和模拟执行质量，以及何时才允许进入“小资金 live pilot 设计”阶段。

本文只定义证据和评审门槛。满足门槛不等于授权真实交易，也不证明策略未来有效；live pilot 的账户隔离、交易授权、费用、公司行动、资金流、风控和紧急停止仍需单独设计和批准。

## 冻结范围

forward 观察开始前必须冻结：

- `strategy_name` 和 `strategy_version`。
- 5-sleeve universe 及 symbol 映射。
- rebalance calendar 和决策时点。
- risk budget、target weight 和取整规则。
- baseline 定义。
- 成本与滑点报告口径。
- shadow account seed。

观察期间发现问题可以修复数据或工程缺陷，但不能静默改变策略规则。任何会改变历史决策的修复必须升级 strategy version，并从新版本起点单独统计；旧版本证据保留。

## 观察窗口

- 最低窗口：3 个完整月度调仓周期，用于检查工程与运行纪律。
- 目标窗口：6 个完整月度调仓周期，用于形成初步 forward 行为证据。
- 少于 3 个周期时，只能报告运行状态，不做 live pilot 准入判断。
- 3～5 个周期可以拒绝或延长观察，但不能以短期收益优秀为由提前晋级。

一个周期从当期 frozen target weight 生效开始，到下一期 confirmed plan 接替前结束。被取消的计划仍属于审计样本，但不算作完成的调仓周期。

## 证据维度

### 1. 数据可靠性

每个周期记录：

- 应有和实有 observation 交易日数。
- ETF 收盘价、交易日历、宏观和利率字段的缺失与 stale 情况。
- `complete`、`partial`、`unavailable` 状态分布。
- 国债曲线主源和 fallback 的实际命中情况。
- 数据修复、补跑和下游重建次数。
- 是否出现发布日期错位、未来数据或版本混用。

任何确认的未来数据使用都直接判定本次观察窗口无效；修复后必须重新开始受影响版本的观察。

### 2. 决策可重复性

每个调仓日保存并核对：

- frozen 输入能否重建相同 regime、risk budget 和 target weight。
- robustness report 是否在人工 decision 前生成。
- `partial` 降级原因是否在计划摘要中可见。
- operator 的确认、取消或延迟原因。
- 是否存在看到后续收益后修改参数或输入的行为。

### 3. 模拟执行质量

每个 plan 记录：

- planned quantity、filled quantity 和 fill ratio。
- 计划估值价、成交均价及方向一致的价格偏差。
- 从确认到首笔成交、全部成交的时间。
- 未成交、部分成交和取消原因。
- 计划后现金比例、实际权重和目标偏离。
- 按当前零费用假设计算的结果，以及独立成本敏感性结果。

模拟成交价格必须来自当时可观察、可追溯的来源。不得使用事后最优价、日内最低买价或最高卖价。

### 4. Forward 表现

报告至少包含：

- shadow 和 inverse-vol baseline 的区间及累计收益。
- 相对收益、最大回撤、年化波动率和换手率。
- sleeve contribution 和无法归因的成交日。
- 月度收益及完整性标记。
- gross、固定费用和滑点情景下的结果。

收益结果只用于判断行为是否与设计一致，不设置“累计收益必须为正”或“必须跑赢 baseline”的短窗口硬门槛。短期超额收益不能覆盖数据时点、执行纪律或审计问题。

### 5. 运维纪律

记录：

- 每日任务按时完成率。
- 阻断、失败、补跑和人工介入次数。
- observation 缺口的发现与修复时长。
- 告警是否具有可操作信息。
- 是否发生 artifact 覆盖、重复写入或无法重建。

## 周期复盘

每个周期结束后生成一份 repo-visible post-mortem，至少回答：

1. 本期模型在当时看到了哪些数据，哪些字段降级？
2. 目标权重相对上期为什么变化？
3. 人工 decision 是否偏离计划，原因是什么？
4. 模拟成交与计划价格、数量和时间相差多少？
5. 实际权重漂移和收益主要来自哪里？
6. baseline 对比是否使用相同且完整的日期？
7. 本期发现的是数据缺陷、工程缺陷、执行假设问题，还是策略假设问题？
8. 是否需要修复、延长观察、升级 strategy version 或停止候选策略？

## 准入判定

评审结果只有三种：

| 结果 | 含义 |
| --- | --- |
| `CONTINUE_SHADOW` | 证据不足或存在可修复问题，保持当前 shadow 阶段 |
| `STOP_OR_REDESIGN` | 出现数据泄漏、不可重建、持续失真或策略边界失效 |
| `ELIGIBLE_FOR_LIVE_PILOT_DESIGN` | 工程和运行证据达到门槛，可以开始单独设计 live pilot |

`ELIGIBLE_FOR_LIVE_PILOT_DESIGN` 不是实际资金授权。

## Live Pilot 设计准入门槛

必须同时满足：

### 工程门槛

- 至少 3 个、目标 6 个完整调仓周期。
- 所有应有交易日都有 observation，或每个缺口都有明确原因、修复记录且最终补齐。
- 不存在未来数据、release-date 错位或 strategy version 混用。
- 所有 frozen 输入都能确定性重建对应计划和账户状态。
- 没有 artifact 覆盖、重复业务键冲突或无法解释的现金/持仓差异。

### 运行门槛

- 每个计划都有明确的人工 decision。
- 每笔模拟成交都有当时可获得的价格来源和时间戳。
- 部分成交、未成交、取消和异常均能由 post-mortem 解释。
- 任务与告警流程至少经历一次失败恢复演练。
- operator 未绕过阻断校验，也未因短期表现临时改规则。

### 研究门槛

- robustness report 与 forward report 使用冻结且一致的策略定义。
- 成本和滑点后结果没有暴露出实现假设与可交易现实的系统性矛盾。
- 相对 baseline 的差异能由权重、换手和 sleeve contribution 解释。
- 若策略未稳定跑赢 baseline，结论明确保持 research-only，不包装为已验证 alpha。
- 国债曲线或其他 confirmatory 数据长期缺失时，已明确其对决策的实际影响；不能只用 `partial` 标签掩盖结构性缺口。

## 直接否决条件

以下任一情况成立时不得进入 live pilot 设计：

- 使用未来数据或事后修订值生成当期决策。
- 同一 strategy version 在观察期间改变规则、参数或 universe。
- 无法从 frozen artifact 重建持仓、现金、净值或计划。
- 模拟成交使用不可追溯或明显事后选择的价格。
- 多次出现数据缺失却通过前值、零值或手工修改绕过校验。
- 需要持续人工编辑 artifact 才能维持流程。
- 真实执行所需的费用、最小交易单位、流动性或账户约束与 Stage N/O 假设明显冲突。

## Live Pilot 设计的后续范围

通过准入后，只允许启动一份新的设计文档，至少覆盖：

- 独立小资金账户与资金上限。
- 人工确认、双重检查和交易权限。
- 券商订单、成交回报与 reconciliation。
- 佣金、滑点、分红、拆分和基金份额折算。
- 现金流和账户外部变动。
- 单 ETF、单 sleeve、组合和单日风险上限。
- 数据异常、订单异常和净值异常的 emergency stop。
- shadow 与 live 并行对账及退出条件。

在该设计通过前，仓库不得增加真实下单入口或 credential 配置。

## 完成标准

- 每个 forward 周期都有完整、可追溯的周期复盘。
- 3～6 个周期汇总报告覆盖数据、决策、执行、表现和运维五个维度。
- 评审结果能够由上述门槛机械复核，而不是由短期收益主观决定。
- live pilot 的准入与真实资金授权保持明确分离。
